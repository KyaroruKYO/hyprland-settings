## Enablement architecture
- Writability mode: gated high-risk writable.
- Why these are gated high-risk writes: the enabled rows are display/render, cursor/input, and debug/crash settings; they require persisted recovery proof and cannot use the ordinary low-risk write path.
- How the production gate is enforced: `SAFE_WRITABLE_ROWS` now includes the 62 rows, but `write_flow` requires `HighRiskProductionGateProof` for gated high-risk rows before scalar writing.
- How missing gate proof is rejected: the default apply path supplies no high-risk gate proof and rejects these rows with a high-risk production gate failure.
- How rollback is required: accepted production gate proof must include backup proof, rollback proof, and parser reread proof through the persisted recovery scaffold.
- How timeout/no-confirmation is handled: timeout or no confirmation selects rollback and rejects keep/apply.
- How UI warnings are required: high-risk write policies project watchdog/recovery warning metadata for the enabled rows.
- How real config mutation was avoided in tests: all write proof uses temp config paths and recovery plans; real config paths are not used.
